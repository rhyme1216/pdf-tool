1. 产品形态与总体目标
1.1 目标
实现一个批量“文档/网页/文本 → PDF”的转换工具，要求：

转换结果 排版尽量不丢失，视觉效果几乎一致（对 Office 文档尽量贴近 Office 的打印/导出效果）。
字体兜底完善，不能出现黑色方块/乱码。
输出 PDF 必须嵌入字体（至少对工具可控链路做到强约束；并提供检测/告警）。
1.2 交付形态（前后端分离、无服务器）
交付为桌面应用（Windows + macOS 都要）。
前后端分离：
前端：Web UI（运行在桌面 App 内）
后端：本地服务层（负责文件处理与转换）
不依赖公网后端服务器；所有转换在用户本机完成（离线可用）。
需要支持“在线构建/打包”能力：通过 CI（如 GitHub Actions）一键产出 Windows 安装包与 macOS App。
2. 输入/输出与文件处理规则
2.1 输入方式
用户在 UI 中上传一个压缩包（zip），zip 内包含文件与目录结构。
工具需要在本地解压该 zip，并按原目录结构处理文件。
2.2 支持的待转换文件类型（枚举）
Word：.doc、.docx（必须支持）
Excel：.xls、.xlsx（必须支持）
HTML：.html
Markdown：.md
Text：.txt
2.3 输出目标类型
目标类型固定为：PDF
2.4 批量转换范围
对 zip 解压后的同一目录树下所有上述类型文件进行批量转换（递归扫描）。
其他文件类型不处理，原样保留在输出包中（是否保留可配置；默认建议保留）。
2.5 输出目录结构与命名
输出保持原 zip 的目录结构不变。
单文件输出：
成功：生成与原文件同名的 .pdf，放在原文件所在目录。
成功后：删除该成功文件的原始源文件（仅删除解压目录内的源文件；不删除用户原始 zip）。
Excel 多 sheet 拆分（可选）：
若开启“按 sheet 拆分”：
输出为多个 PDF：原文件名_SheetName.pdf
SheetName 需做文件名安全清洗（非法字符替换为 _ 等）
若清洗/重名导致冲突，依次追加  (1),  (2)…保证唯一性
2.6 失败处理与日志
任一文件转换失败时：
不中断整个批量任务，继续转换后续文件。
失败文件：在输出包中保留原始源文件（不删除）。
输出 zip 根目录生成 error.log：
记录每个失败文件的相对路径、文件类型、使用的引擎、错误信息（可含堆栈/退出码/时间戳）。
3. 转换渲染一致性与引擎策略
3.1 一致性目标
“几乎一致”：不要求像素级完全一致，但要求分页、表格、图片、基本布局尽量保持，不能明显错位、缺内容。
Office 文档转换以“尽可能接近 Office 打印/导出效果”为目标。
3.2 引擎策略（按平台）
Windows
Word/Excel：优先使用 Microsoft Office（本机安装）导出 PDF（高保真优先）。
若 Office 不可用或导出失败：fallback 到内置 LibreOffice headless转换。
HTML/MD/TXT：使用Chromium headless 打印生成 PDF。
macOS
Word/Excel：使用内置 LibreOffice headless转换。
HTML/MD/TXT：使用Chromium headless 打印生成 PDF。
3.3 并发与稳定性
默认不建议并发：转换任务按队列串行执行，优先稳定与一致性。
4. 具体格式的特殊规则
4.1 Excel 多 sheet 处理（可选项）
提供 UI 选项：是否“按 sheet 拆分成多个 PDF”。
关闭时：整个工作簿导出为一个 PDF（所有 sheet 依次输出）。
4.2 Markdown
Markdown 渲染引擎不做强约束（实现即可），但需提供稳定的默认样式（CSS）以保证输出可读。
4.3 TXT
规则：每 1000 字符硬分页（按字符数切分分页，不依赖自动分页算法）。
输出 PDF 中需保持文本可读（建议等宽字体、合理行距、自动换行）。
5. 字体兜底与国际化要求
5.1 语言覆盖
至少需保证以下语言/文字不出现黑块/乱码：

中文、英文
泰语、阿拉伯语
葡萄牙语、印尼语、越南语、匈牙利语、马来语
5.2 字体要求
必须有字体兜底逻辑：当源文档字体缺失/不支持时，用内置字体替代，尽量避免渲染异常。
输出 PDF 必须嵌入字体（至少对 Chromium 打印链路强制；对 Office/LibreOffice 链路尽最大努力并提供检测/告警）。
5.3 开箱即用
LibreOffice 采用 内置打包方案（不要求用户自行安装）。
字体包同样随应用内置（按需要取舍体积，但必须覆盖上述语言）。
6. 桌面应用与打包要求
必须提供：
Windows 安装包（EXE/MSI 均可）
macOS 应用包（DMG/APP）
支持通过 CI 进行“在线打包构建”（例如 GitHub Actions 自动产物发布）。
